     7→That said, I have some concerns about the current scoring method.
     8→The current scorer estimates memory and params from the ONNX graph, and then computes the score from those values. However, this makes it possible to optimize not only for simple and efficient task solvers, but also for gaps in the scorer's static analysis logic.
     9→In fact, I have already confirmed that it is possible to obtain 25 points on an arbitrary task under the current scorer . I am intentionally not describing the exploit details here, but the important point is that this is not just a theoretical concern.
    10→This means that a theoretical 25 * 400 = 10000 score can be reachable through scorer-specific behavior. This is different from the original goal of solving ARC tasks well and efficiently in ONNX.
    11→Another concern is that, with the current static scoring approach, a participant who finds an exploit could keep it private and use it only for the final submission. It would also be possible to update the score gradually in order to maintain the leaderboard lead. This affects not only fairness between participants, but also the reliability of the leaderboard.
    12→As I understand it, recalculation for the first-place duration prize starts from May 6. If scorer exploits can move the leaderboard significantly, the operations team may need to spend more and more effort on remeasurement, investigation, and scorer fixes. From the participant side as well, attention may shift from improving ARC task solutions to searching for the next scorer exploit, making it harder to focus on the original purpose of the competition.
    13→Of course, individual scorer issues can be fixed. However, ONNX has many complex aspects: shape inference, constant folding, runtime behavior, operator semantics, and tooling differences. It seems very difficult to estimate memory and compute perfectly, statically, and fairly. Even if one exploit is fixed, another one may be found quickly. If the leaderboard changes significantly each time, the competition becomes unstable.
    14→For this reason, I would like to propose a simpler and more deterministic scoring method: use the submitted ONNX file size.
    15→For example, each task could be scored as follows:
    16→score = max( 1 . 0 , 25 . 0 - log(max( 1 . 0 , onnx_file_size_bytes)))
    17→Here, log is the natural logarithm.
    18→This has several clear advantages:
    19→It is very simple to implement.
    20→It depends much less on scorer internals.
    21→It is less affected by ONNX tooling, runtime, or profiler differences.
    22→It evaluates the submitted artifact itself, so the exploit surface is smaller.
    23→It matches the NeuroGolf goal of building short, simple, and efficient ONNX solvers.
    24→As in last year's CodeGolf competition, optimizing byte size is not just superficial compression. It often leads participants to discover simpler and more efficient task-solving ideas and representations. I think the same applies here: minimizing ONNX file size would guide participants toward concise and essential ONNX solutions.
    25→Of course, file size is not a perfect proxy for runtime memory or compute. A small graph can still create large intermediate tensors. However, the more complex the scorer's memory and compute estimation logic becomes, the more that estimation logic itself becomes the optimization target. For a competition metric, a simple and stable proxy may be preferable to a more sophisticated but exploitable estimator.

The Neurosynthetic Research Institute · Research Prediction Competition · 12 days to go Join Competition more_horiz The 2026 NeuroGolf Championship Design the smallest neural networks to solve ARC-AGI image transformations The 2026 NeuroGolf Championship Overview Data Code Models Discussion Leaderboard Rules souldrive · 270th in this Competition · Posted 21 days ago arrow_drop_up 1 more_vert Compile, Don't Train: the opset-10 ONNX trick that scores ~2× on held-out TL;DR. In NeuroGolf the per-task score rewards small, simple nets that nail the held-out examples. We stopped training per-task CNNs and started compiling reference ARC programs into opset-10 ONNX. Compiled rules generalize (~0.93–1.0 held-out) and are tiny, so they beat trained memorizers (~0.42) by roughly 2× of public value. Below: how to reason about the metric, an empirical "scoring is fractional" finding, the reusable opset-10 golf tricks, a debugging method, and an honest list of what this can't do.
    46→▶ Runnable companion notebook (every trick below as a real, verified opset-10 graph): https://www.kaggle.com/code/souldrive/compile-don-t-train-onnx-golf-that-scores-2x
    47→The metric, restated as a strategy
    48→Per task, score ≈ (25 - ln(mem + params)) × (fraction of held-out scored examples predicted exactly) .
    49→Two forces fall out of that single line:
    50→Smaller is more. Because it's 25 - ln(mem + params) , every byte you shave is logarithmic points back. "Memory" here is mostly intermediate tensor size , not weights — that distinction is where most of the points hide (see Memory Golf below).
    51→Right beats close, but close beats zero. The second factor is a fraction , so a net that implements the true rule on more held-out examples scores strictly more.
    52→That second point reframes the whole competition. If you believe scoring is binary (exact-on-everything or zero), you train a CNN to memorize your visible examples and hope. If scoring is fractional , you should instead implement the rule that produced the examples — because the rule is what transfers to the held-out split.
    53→The empirical finding: scoring is FRACTIONAL, not all-or-nothing
    54→We didn't assume this — we measured it. We took several "near-miss" transpiles (programs exact on ~90–99% of our examples, not 100%) and submitted them on empty task slots. Public score went up ~+25 .
    55→Under all-or-nothing scoring that change is +0 . The only model consistent with "+25" is:
    56→task_points = base_points × (held- out exact fraction )
    57→This is the single most useful thing we learned, because it tells you what to optimize : not "fit my visible examples" but "implement the rule that generates them." Memorizers cap out near their visible-set fraction on held-out data; rule-implementers ride the fraction up toward 1.0.
    58→A public ARC DSL ships ground-truth solve_<id>(I) programs for many tasks. Transpile those into compact ONNX and you get both metric factors at once — tiny and correct:
    59→The mental model that makes this click: a trained CNN approximates the input→output map from a few points; a compiled program is the map. Held-out is exactly where that difference shows up.
    60→One honest caveat up front, because it shapes how you should use all of this: a fairly complete transpiler tops out around ~3400 local , which is below this competition's Bronze cutoff . So treat this as a strong rank-climb plus a reusable engineering technique — not a guaranteed-medal method.
    61→The reusable part: opset-10 ONNX-golf tricks
    62→This is the section to steal even if you never touch ARC. opset-10 is deliberately spartan — no Loop , no Unique , no NonZero , no CumSum , and Min / Max are float-only — so the "obvious" implementations are gone and you express everything as fixed-shape tensor algebra. These patterns transfer to any compact-ONNX / net-golf target.
    63→1. Enumerate components without Loop : the fixed-K peel
    64→You can't iterate an unknown number of objects. Pick a budget K and peel the highest remaining component K times:
    65→remaining = labels # int/ float grid , 0 = background
    66→for k in range (K): # K is a compile - time constant -> fully unrolled
    67→comp = Greater(remaining, mx - 0.5 ) * Greater(mx, 0.5 ) # the top object (empty if none)
    68→remaining = remaining * ( 1 - comp) # erase it, continue
    69→K iterations yield up to K objects in descending-label order; iterations past the last object produce empty masks that fold away. Pick K = your worst-case object count.
    70→2. Connected-component labeling via unrolled max-propagation
    71→To get those labels, seed each foreground cell with a unique id, then propagate the neighbor-max for H + W steps — BFS flood unrolled to its worst-case diameter:
    72→ids = unique_id_per_cell * foreground # e.g. row *W + col + 1 , zeroed on background
    73→nbr = max (ids, shift_up(ids), shift_down(ids), shift_left(ids), shift_right(ids))
    74→ids = nbr * foreground # re-mask so labels never leak into background
    75→After convergence every cell carries its component's max id — a stable label you feed straight into Trick 1. Shifts are Pad + Gather (Trick 6), so everything stays static-shape.
    76→3. Prefix sums without CumSum : triangular MatMul
    77→A prefix sum along an axis is exactly a multiply by a triangular ones-matrix:
    78→L = lower_triangular_ones(N, N) # constant initializer, L[i,j] = 1 if j <= i
    79→Use the strict ( j < i ) variant for exclusive sums. This is the workhorse for compaction : prefix-sum a 0/1 keep-mask to get each kept element's destination index.
    80→4. Variable-length outputs (compress / dedupe / split / crop) via a permutation MatMul
    81→compress / dedupe / split / crop all produce a variable number of lines — normally a NonZero + Gather job you can't do here. Instead build a 0/1 permutation matrix that scatters kept lines to the top-left, and apply it with MatMul :
    82→keep = (row is kept) # 0/1 vector, length H
    83→dest = strict_prefix_sum(keep) * keep # Trick 3: kept row i -> position dest[i]
    84→R = on ehot_scatter(dest, keep) # R[dest[i], i] = 1 for kept rows
    85→rows_compacted = MatMul(R, grid) # rows pulled to the top; rest zero
    86→# columns: build Cm the same way and do MatMul(grid, Cm)
    87→Rows use a left-multiply, columns a right-multiply. The output stays a static H×W ; kept content lands top-left, the rest is zero.
    88→5. Canvas mode: keep the output shape STATICALLY inferable
    89→This one is load-bearing. The scorer sizes your net via shape-inference, so your output shape must be statically inferable. A dynamic Slice makes the shape runtime-only → the net is unscoreable .
    90→So for variable-size outputs, always emit onto a fixed 30×30 canvas, content packed top-left, with a runtime keepmask that zeros everything beyond the runtime (H, W) extent:
    91→canvas[30, 30] # static shape -> shape-inference is happy
    92→keep = (row_idx < H) * (col_idx < W) # runtime mask from the data, broadcast to 30x30
    93→Variable size lives in the values , never in the shape .
    94→Corollary: subgrid / crop is two static-shape Gather s , never a dynamic Slice . The index values can be data-dependent; the index shape is static, so the output shape is too — and this is also how every "shift" in Trick 2 is built.
    95→6. The all-zeros profiling pass → clamp every index
    96→The scorer profiles each net on an all-zeros input to measure cost. That's a trap: with an empty grid your masks are empty, your "first kept index" is garbage, and a Gather happily reads out of bounds . Clamp every data-derived index into [0, N-1] before you gather. opset-10 Min / Max are float-only, so clamp in float and cast back:
    97→Do this on every index that derives from data, not just the obvious ones. The all-zeros pass will find the one you forgot.
    98→7. Memory golf: trace on the compact frame, then Pad back
    99→Points scale with 25 - ln(mem + params) , and most of mem is intermediate tensor size, not weights. So the biggest lever is often: don't run your pipeline on the full 30×30 . Crop the input to its content bounding box B×B (two Gathers, Trick 5), run the entire trace on the compact frame, then Pad back to 30×30 :
   100→B = content_extent(grid) # bounding box of non-background
   101→core = Gather2D(grid, B) # run everything on B x B
   102→out = Pad(core, to= 30 x30) # restore static canvas at the end
   103→On one heavy net this dropped "memory" from ~59.7M → ~1.6M — a large, basically-free gain, because the cost is logarithmic in mem + params .
   104→Two strict-equality footguns (these cost me real submissions)
   105→These aren't opset tricks — they're equality-contract traps. Both produce nets that "look right" and fail the strict compare.
   106→Refill channel 0. The harness compares the full one-hot , and colour 0 means channel-0 = 1 , not "all channels 0". Any cell you zero across all channels reads as invalid , not colour-0, and fails strict equality even though it renders identically:
   107→ch0 = 1 - ReduceSum(onehot, axis=channels) # cells now empty
   108→onehot [..., 0 ] = onehot[..., 0 ] + ch0 # refill background so 0 == channel- 0 = 1
   109→Clip fills to the content extent. A flood/fill that overshoots paints the zero-padding outside (H, W) → off-grid colours that fail the strict 30×30 compare. Clip every fill to the real extent (the Trick 5 keepmask does it).
   110→The F-lever: one substrate, a whole family of tasks
   111→Once you can enumerate objects (Trick 1), one substrate unlocks a family of tasks: a general higher-order sfilter /apply over enumerated objects .
   112→objects = peel_K(mask) # Trick 1
   113→keep = [ coerce_bool(predicate(o)) for o in objects ] # any per-object test
   114→result = union( o * keep_o for o in objects ) # gate + union survivors
   115→Evaluate any predicate per single-component object (size, colour count, symmetry, touches-border, unique-colour, …), coerce to a keep-bool, gate the cells, union the survivors. Most "select the objects that …" tasks collapse into picking the predicate. Build the substrate once; reuse it everywhere.
   116→Debugging: partial-program verify
   117→When a transpiled program is wrong but you don't know where , build synthetic targets from the reference program's intermediate state :
   118→Run the reference solve_<id>(I) up to intermediate variable x_k .
   119→Make a synthetic example whose target output is x_k .
   120→Run your transpiled graph truncated to the same point and compare.
   121→The first x_k that diverges is the exact statement your transpiler got wrong. This turns "the net is wrong somewhere" into "node 37 mismatches" — the difference between an afternoon and five minutes.
   122→Companion heuristic: the compiles-but-wrong per-example diff is your highest-yield signal. A net that compiles but misses a few examples almost always encodes one concrete bug or a single semantic gap, and the per-example diff points right at it. Chase those before anything else.
   123→Honest limits — what this cannot do
   124→I'd rather you trust the rest of the post, so here's the unflattering half:
   125→The DSL has to actually reproduce the scored examples. The transpiler can only bank a task where the reference program matches the scored arc-gen examples (~306/400 for us). The rest are arc-quirk : the reference disagrees with the generator (classic case — the DSL keeps a border the scored output strips). A perfect transpile of the wrong rule is still wrong — unbankable, so detect and skip rather than burn hours.
   126→Order-dependent selection is fragile. Ops like "take the first / extract over a set" depend on iteration order, and our connected-component ordering differs from the reference's frozenset order. A few tasks mismatch purely on order. We skip those rather than ship a wrong net.
   127→There's a ceiling, and it's below Bronze (~3400 local, as noted above). This is a rank-climb and a reusable technique, not a guaranteed medal — I'd rather say that plainly than oversell it.
   128→The exact-verify gate is the unsung hero of the honest version: every banked net is verified exact on our examples before it ships, so the failure mode is "we skip a task," never "we silently submit a wrong net."
   129→Takeaways you can reuse tomorrow
   130→On any net-golf metric, first decide if scoring is fractional — submit a near-miss on an empty slot and watch the delta. It changes your entire strategy.
   131→If it's fractional, implement rules, don't memorize examples — rules ride the held-out fraction toward 1.0.
   132→The opset-10 patterns ( fixed-K peel, max-propagation labeling, triangular-MatMul prefix sums, permutation-MatMul variable outputs, static canvas + keepmask, clamp-every-index, bounding-box memory golf ) port to any compact-ONNX target.
   133→Build one higher-order substrate (the F-lever) and reuse it across a whole task family.
   134→Verify exact before you ship , and partial-program-verify to localize bugs fast.
   135→If you're new here: you don't need to train anything to make progress. Pick one task with a known reference program, hand-compile it to opset-10 using the tricks above, verify it exact, and submit it on an empty slot. Watch the score move. That single loop taught us more than any amount of model tuning.
   136→▶ Want to run it? Every trick above is a real, executable opset-10 graph in the companion notebook: https://www.kaggle.com/code/souldrive/compile-don-t-train-onnx-golf-that-scores-2x — fork it, run it, and the vmirror + compress demos verify exact in seconds.
   137→Happy to go deeper on any of these — the permutation-MatMul and the canvas/keepmask shape trick especially have details I compressed here. A couple of open questions I'd genuinely love help with:
   138→Better tricks for order-stable component enumeration in opset-10? That's our biggest remaining source of skips.
   139→Does "compile, don't train" generalize to other net-golf metrics you've seen? I suspect it does, well beyond ARC.
