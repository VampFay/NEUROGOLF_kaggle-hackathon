  2632тЖТ## Solution Summary
  2633тЖТ
  2634тЖТ 
  2635тЖТ
  2636тЖТ## Abstract
  2637тЖТ
  2638тЖТ This paper presents the Omni-ARC approach to the 2024 Abstraction and Reasoning Challenge (ARC), focusing on data efficiency and the abstraction aspect of the challenge. The core of this solution lies in training a transformer to perform multiple tasks related to ARC problems, going beyond the challenge's primary objective of predicting outputs from input-output examples. This multi-task learning approach aims to develop a robust representation of the ARC problem space.
  2639тЖТ
  2640тЖТ Omni-ARC leverages publicly available LLMs, fine-tuning them on various ARC-related tasks, including output prediction and input distribution learning, using augmented data from multiple ARC datasets. The solution incorporates test-time fine-tuning to enhance accuracy on the private test set, achieving a score of 40 on the public leaderboard, securing 3rd place in the challenge.
  2641тЖТ
  2642тЖТ Further analysis demonstrates the potential of training models to verify output correctness and select the most probable solution from multiple options, paving the way for future research in integrating reasoning capabilities into the Omni-ARC framework.
  2643тЖТ
  2644тЖТ## Intro
  2645тЖТ
  2646тЖТ## Abstraction and Reasoning Challenge
  2647тЖТ
  2648тЖТ There are two different definitions of artificial intelligence. The first says that artificial intelligence is
  2649тЖТthe science of creating machines that can do the tasks that humans can do. According to this definition
  2650тЖТwe would be very close to artificial general intelligence (AGI) because systems like ChatGPT can do many
  2651тЖТtasks that only humans were able to do before such as solving math problems, answering all kind of questions,
  2652тЖТimproving the style of some text... Many AI researchers believe that scale is all we need, and simply
  2653тЖТscaling the models and the data will lead to AGI.
  2654тЖТ
  2655тЖТ But there is also another view championed by Fran├зois Chollet that says that skill is not intelligence,
  2656тЖТthat intelligence is the ability to handle novelty and learn new skills.
  2657тЖТ
  2658тЖТ 
  2659тЖТ The intelligence of a system is a measure of its skill-acquisition efficiency over a scope of tasks, with respect to priors, experience, and generalization difficulty.
  2660тЖТ
  2661тЖТ 
  2662тЖТ According to this definition we are far from AGI because the accuracy of the current deep learning models
  2663тЖТon some task does not depend on the complexity of the task, but on the familiarity of the model with the task.
  2664тЖТ
  2665тЖТ To spark research into this view of intelligence Chollet created the Abstraction and Reasoning Corpus (ARC)
  2666тЖТand so far it has resisted the test of time. Whereas LLMs are saturating all kind of publicly available
  2667тЖТbenchmarks, they still do very poorly on the ARC dataset. This can be explained by two big reasons:
  2668тЖТ
  2669тЖТ 
  2670тЖТ
  2671тЖТ- The test dataset is private 
  2672тЖТ
  2673тЖТ- All the problems in the test dataset are novel 
  2674тЖТ 
  2675тЖТ In 2024 Chollet joined forces with Mike Knoop to launch the ARC Prize 2024, with a total prize pool of 1M$.
  2676тЖТThe goal was to rise awareness of the unbeaten ARC and to increase the number of people working to solve it.
  2677тЖТ
  2678тЖТ## Motivation of my approach
  2679тЖТ
  2680тЖТ In the ARC challenge we have to learn a transformation rule given a few
  2681тЖТhigh-dimensional pairs of input and output images. The images can have a size of up to 30x30 pixels
  2682тЖТand each pixel can take 10 different colors. The images are not as complex as real world images, but nevertheless
  2683тЖТthey are high dimensional data.
  2684тЖТ
  2685тЖТ## How can we learn from few high-dimensional examples?
  2686тЖТ
  2687тЖТ 
  2688тЖТ
  2689тЖТ To solve each ARC problem we have to find the right representation of the data. When humans solve the
  2690тЖТtasks, the biggest challenge is to find the right perspective to look at the problem. Once we have the right
  2691тЖТperspective of the data the ARC problems are trivial to solve.
  2692тЖТ
  2693тЖТ The right representation of the data allows to decrease the dimensionality of the data and makes
  2694тЖТpossible to learn the transformation from very few examples.
  2695тЖТ
  2696тЖТ 
  2697тЖТ
  2698тЖТ## How can we learn a good representation of the ARC problems?
  2699тЖТ
  2700тЖТ If we train a model to do tasks that require a good representation of the data, it's likely that the
  2701тЖТmodel will internally develop the required representation.
  2702тЖТ
  2703тЖТ My insight was that we could use the ARC problems in many different ways to learn that representation,
  2704тЖТnot just in the original proposed task that asks to generate the output for an image given a few input-output pairs.
  2705тЖТThe next section shows the different ways to use the ARC problems.
  2706тЖТ
  2707тЖТ## Omni-ARC: Training a single model to do multiple ARC-related tasks
  2708тЖТ
  2709тЖТ 
  2710тЖТ 
  2711тЖТ 
  2712тЖТ examples + input -> output 
  2713тЖТ examples -> code 
  2714тЖТ code + input -> output 
  2715тЖТ 
  2716тЖТ 
  2717тЖТ 
  2718тЖТ 
  2719тЖТ 
  2720тЖТ 
  2721тЖТ 
  2722тЖТ 
  2723тЖТ 
  2724тЖТ inputs -> input 
  2725тЖТ code -> inputs 
  2726тЖТ inputs -> code 
  2727тЖТ 
  2728тЖТ 
  2729тЖТ 
  2730тЖТ 
  2731тЖТ 
  2732тЖТ 
  2733тЖТ 
  2734тЖТ 
  2735тЖТ 
  2736тЖТ
  2737тЖТ- examples + input -> output . The original task of the ARC dataset. 
  2738тЖТ
  2739тЖТ- inputs -> input . Generating new inputs requires to understand the distribution of the grids. It could also be done with the outputs, that should also follow some distribution. 
  2740тЖТ
  2741тЖТ- examples -> code . This is the approach used by Ryan Greenblat with GPT-4o, generate code to solve the task given some ARC examples. 
  2742тЖТ
  2743тЖТ- code + input -> output . This is equivalent to the first task, but instead of giving examples as input, it gives the code definition of the problem. 
  2744тЖТ
  2745тЖТ- code -> inputs . Each input to a task follows some distribution, given a description of the
  2746тЖТ distribution the model should be able to generate samples of that distribution. 
  2747тЖТ
  2748тЖТ- inputs -> code . We could also do the opposite task, given some inputs write code to generate that distribution. 
  2749тЖТ
  2750тЖТ- examples + input + output -> is the output correct? . It is possible to train the model to verify wether a proposed output is correct. 
  2751тЖТ
  2752тЖТ- examples + input + output options-> select the correct output . We can train a model to select the correct output between multiple options. 
  2753тЖТ 
  2754тЖТ All the listed tasks require that the model learns some useful representation of the ARC image. The idea
  2755тЖТbehind the Omni-ARC approach is to train a single model to do all the tasks, with the expectation that a shared
  2756тЖТrepresentation across all the tasks will generalize better than training the model to do a single task.
  2757тЖТ
  2758тЖТ 
  2759тЖТ
  2760тЖТ Omni-ARC, a single model that does all the ARC-related tasks (and it has a very cool logo) 
  2761тЖТ
  2762тЖТ## Prior work
  2763тЖТ
  2764тЖТ## MindsAI
  2765тЖТ
  2766тЖТ The most relevant prior work is the information given by the MindsAI team about they approach. On
  2767тЖТinterviews they have told that they biggest contribution is to do test-time fine-tuning. There was little
  2768тЖТinformation but enough to make educated guesses and replicate their results:
  2769тЖТ
  2770тЖТ 
  2771тЖТ
  2772тЖТ- Test-Time Augmentation to solve ARC, interview with Jack Cole 
  2773тЖТ
  2774тЖТ- Machine Learning Street Talk | Chollet's ARC Challenge + Current Winners 
  2775тЖТ 
  2776тЖТ 
  2777тЖТ Our ARC solution stands out due to several key elements. Firstly, we fine-tune models on synthetic and augmented data. Secondly, we employ test-time fine-tuning. Lastly, we have developed an approach called AIRV (augment, inference, reverse augmentation, and vote), which is analogous to test-time augmentation. These innovations are crucial, as transformer models perform relatively poorly on ARC without them.
  2778тЖТ
  2779тЖТ 
  2780тЖТ I could summarize my solution as an extension of the MindsAI approach, in addition to the three points
  2781тЖТcited above my approach trains the model to perform more tasks around the ARC data. That way we can
  2782тЖТimprove the data efficiency of the system and get better results for the same amount of data.
  2783тЖТ
  2784тЖТ## Ryan Greenblatt
  2785тЖТ
  2786тЖТ 
  2787тЖТ I recently got to 50%1 accuracy on the public test set for ARC-AGI by having GPT-4o generate a huge number of Python implementations of the transformation rule (around 8,000 per problem) and then selecting among these implementations based on correctness of the Python programs on the examples (if this is confusing, go to the next section)2. I use a variety of additional approaches and tweaks which overall substantially improve the performance of my method relative to just sampling 8,000 programs.
  2788тЖТ
  2789тЖТ 
  2790тЖТ The approach taken by Ryan Greenblatt was very inspiring because he didn't fine-tune any model for the ARC challenge.
  2791тЖТ
  2792тЖТ I tried to emulate his approach using open and smaller LLMs with the aim to combine it with the MindsAI
  2793тЖТapproach but my efforts failed. However I believe that if I devote more work to this approach it might work.
  2794тЖТ
  2795тЖТ## Approach
  2796тЖТ
  2797тЖТ 
  2798тЖТ
  2799тЖТ The solution on a nutshell:
  2800тЖТ
  2801тЖТ 
  2802тЖТ
  2803тЖТ- Take Qwen2.5-0.5B-Instruct and fine-tune it on publicly available ARC datasets. The model was fine-tuned to: 
  2804тЖТ
  2805тЖТ- generate the outputs for the test samples 
  2806тЖТ
  2807тЖТ- learn the inputs distribution and generate new inputs. 
  2808тЖТ 
  2809тЖТ 
  2810тЖТ
  2811тЖТ- Do test-time fine-tuning with the private test data, only for the task of generating the test outputs. 
  2812тЖТ
  2813тЖТ- Inference with data augmentation, and voting to select the predictions 
  2814тЖТ
  2815тЖТ- Ensemble with the 2020 public solution

  2988тЖТ## Test-time fine-tuning
  2989тЖТ
  2990тЖТ Fine-tuning a model on ARC tasks is not enough to do well on the private test set. By applying test-time fine-tuning we could improve the number of solved problems from 11 to 33 for one of the models that I trained along the challenge.
  2991тЖТ
  2992тЖТ This is my interpretation of the test-time fine-tuning:
  2993тЖТ
  2994тЖТ 
  2995тЖТ
  2996тЖТ- For each test problem that had n train samples, I fine-tuned the model using n-1 train samples and
  2997тЖТ using the remaining sample as a test sample. The selection of the test sample was done randomly on the fly during training. 
  2998тЖТ
  2999тЖТ- I used data augmentation just like in the previous training 
  3000тЖТ
  3001тЖТ- I fine-tuned a model for each of the test problems, so 100 fine-tuned models were generated on each submission. 
  3002тЖТ
  3003тЖТ- I used batch size 1 in the test-time fine-tuning to be able to learn the new problems as fast as possible. 
  3004тЖТ
  3005тЖТ- The model was fine-tuned for ~300 steps on each problem 
  3006тЖТ
  3007тЖТ- Suprisingly the best learning rate for test-time fine-tuning was 8e-5, higher than the one used for
  3008тЖТ training (5e-5). It's very likely that better results could be obtained if more computation was
  3009тЖТ available, training with a slower learning rate, with higher batch size and for longer. 
  3010тЖТ 
  3011тЖТ Due to the limited submission time test-time fine-tuning was only applied to the canonical ARC task
  3012тЖТof predicting the test outputs. But it could also be applied to the task of generating new inputs, or
  3013тЖТto the tasks of verifying the correctness of the outputs.
  3014тЖТ
  3015тЖТ The unusual configuration of training a single model for each task with batch size 1 arose due to the
  3016тЖТlimitations of compute and submission time. It was the configuration that allowed to learn faster the new
  3017тЖТtest problems.
  3018тЖТ
  3019тЖТ## Inference
  3020тЖТ
  3021тЖТ Data augmentation was applied also at inference, and the data augmentation was reverted from the prediction to get the original output. 96 predictions were done for each problem and voting was used to select the most
  3022тЖТpromising predictions. So just like MindsAI's AIRV (augment, inference, reverse augmentation, and vote).
  3023тЖТ
  3024тЖТ Inference was done using a temperature of 0.
  3025тЖТ
  3026тЖТ vLLM was used to generate the predictions. Each fine-tuned model was used to generate predictions for its problem.
  3027тЖТ
  3028тЖТ## Ensemble
  3029тЖТ
  3030тЖТ I ensembled my model predictions with the 2020 solution . Since the 2020 solution only requires CPU, I managed to run
  3031тЖТit on the background while I used the GPU for fine-tuning and inference with my model. I only had to
  3032тЖТbe careful with the RAM usage because both jobs had to share the same memory.
  3033тЖТ
  3034тЖТ The ensemble strategy was very simple, just take the first attempt from each solution.

  3039тЖТ
  3040тЖТ 
  3041тЖТ
  3042тЖТ The same approach (without test-time fine-tuning) could solve 32% of the evaluation tasks, and when using voting with 32 predictions

  3049тЖТ## Learnings
  3050тЖТ
  3051тЖТ## Prompting is not enough, test-time fine-tuning is needed
  3052тЖТ
  3053тЖТ Clearly this competition has shown that LLMs need test-time fine-tuning to do new tasks. Few-shot prompting is not enough for the model to learn novel tasks.

  3142тЖТ## The right model size

  3220тЖТ## Conclusion
  3221тЖТ
  3222тЖТ## Future steps
  3223тЖТ
  3224тЖТ The approach of using a transformer and test-time fine-tuning could likely keep improving and maybe
  3225тЖТsolve the ARC prize if we generate enough synthetic data to densely cover the space of the ARC problems.
  3226тЖТHowever that kind of solution won't give us more knowledge about how to reach AGI. It might be worth
  3227тЖТpursuing that direction just to know where it can get us, but I don't feel it is interesting.
  3228тЖТ
  3229тЖТ On this year competition I have focused on abstraction, on building the best possible representation
  3230тЖТof the ARC problems. But the reasoning part was missing from my solution. When I try to solve the ARC
  3231тЖТproblems I make a hypothesis of the transformation, see if it works on the train data and fix it if
  3232тЖТit doesn't. Finding the solution is typically an iterative process of trial and error.
  3233тЖТ
  3234тЖТ I believe that we can teach a model to reason, just like OpenAI is developing the new o1 models. First we will
  3235тЖТneed to have a model that is able to generate code to solve the problems, otherwise we cannot verify
  3236тЖТthe solution and iterate over the results. Then we will generate many reasoning traces for the training
  3237тЖТtasks and the model could learn to iteratively create a python code solution to the problems. The main
  3238тЖТobstacle that I see to this approach is that it will require a much bigger context size than the current
  3239тЖТMindsAI approach because in addition to the original task the prompt will also have the different
  3240тЖТcode iterations and the outputs. So we could be talking about using up to 50k tokens instead of the
  3241тЖТcurrent 10k tokens. That requires better hardware both for training and inference.
  3242тЖТ
  3243тЖТ Probably the best approach is the one that first tries to generate code to solve the problems and finally
  3244тЖТuses test-time fine-tuning only on the problems that the code approach could not solve. Thus I believe that
  3245тЖТthe Omni-ARC approach, training a single model to do multiple ARC-related tasks, has a great change of
  3246тЖТbeing used in the solution that beats the 85% goal.
