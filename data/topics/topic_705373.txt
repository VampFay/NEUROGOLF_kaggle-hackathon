Is ~6360 the real maximum achievable score for static-shape networks?

I've been studying the scorer and I keep hitting what looks like a hard ceiling, so I wanted to share the reasoning and ask whether I'm missing something.

The score is points = max(1, 25 − ln(macs + memory + params)) per task. Using the official score_network, I measured the cost of a trivial single-node network whose only job is to emit the required [1, 10, 30, 30] output:

bool output → memory = 9000 → 25 − ln(9000) ≈ 15.89 points fp16 output → 18000 → ≈ 15.20 points fp32 output → 36000 → ≈ 14.51 points Since score_network counts every node's output tensor (a 2-node chain measures 72000), the output tensor alone sets a floor: any network that genuinely produces a static [1,10,30,30] result seems bounded near 15.9 points/task, i.e. ~6360 total if every task used a bool output — and lower in practice since many tasks need fp16/fp32 outputs.

Yet the top of the leaderboard sits well above 7000. The only way I can see to beat the output-memory floor is to make the counted output volume smaller than 9000 (e.g. a shape-inference dimension collapsing toward 0).

So my genuine question: is ~6360 the real ceiling for "honest" static-output networks, and is everything above it relying on the output-memory trick — or is there a legitimate technique to get a full [1,10,30,30] result scored below the 9000-byte floor that I've simply missed? Would love to learn if I'm wrong.

for floor-bound tasks, is the general approach to crop the input to a bounded top-left region… how do you size that crop so it safely covers the hidden size range per task?

Yes — and the trick that sidesteps the "how big a crop?" guess is to never bake a per-task crop size into the shape. Two ways:

(a) Compact crop (smallest memory): size B = max(over your visible examples) of the extent + a small margin (I use +2). Hidden inputs come from the same generator, so visible-max + margin covers them in practice — but it's a gamble on the tail.

(b) Static canvas + keepmask (safe for any hidden size): keep the graph at a fixed 30×30, pack content top-left, and multiply by a runtime keepmask (row_idx < H) * (col_idx < W) that zeros everything beyond the runtime (H, W) extent. The output shape stays a static 30×30 (so infer_shapes sizes it and it scores), while the values are correct for whatever the hidden size is. No per-task crop-size to get wrong.

Two things that gate both, learned the hard way:

Use a static-shape Gather, never a dynamic Slice. The index values can be data-dependent, but the index shape must be static — a runtime-shaped Slice makes shape-inference return nothing and the net becomes unscoreable.
Clamp every data-derived index into [0, N-1]. The scorer profiles on an all-zeros input, so masks are empty, the "first index" is garbage, and Gather reads out of bounds. (opset-10 Min/Max are float-only — clamp in float, cast back.)

For floor-bound tasks I'd reach for (b): the keepmask floor is the same [1,10,30,30] you measured, and you stop worrying about the hidden size entirely.
